# CLI Output Formatting Plan

This plan turns the current CLI output review into implementation work. The goal is to move `ts` closer to mature OSS CLI behavior: stable machine output, clean stdout/stderr boundaries, readable human output, and clear help/documentation.

## Goals

- Treat stdout as the command result and stderr as diagnostics, progress, and errors.
- Keep human table output pleasant to read but explicitly non-stable for parsing.
- Make JSON output a documented, tested API.
- Keep command handlers focused on domain behavior, with formatting handled in the output layer.

## Non-Goals

- Do not redesign the command tree.
- Do not break existing JSON shapes unless a versioned migration path exists.
- Do not add decorative terminal output, colors, or interactive prompts as part of this work.

## Phase 1: Document The Current Contract

Create `docs/output-format.md` and document the current behavior:

- Success results are written to stdout.
- Failures are written to stderr.
- `--json` and `--output json` emit one JSON value.
- `--output yaml` emits one YAML value.
- `table` output is human-oriented and not stable for parsing.
- Progress-producing commands stream progress only for human output.

Include JSON examples for high-use commands:

- `ts status`
- `ts channel list`
- `ts client list`
- `ts plugin info`
- `ts connect`
- `ts events watch`

Acceptance criteria:

- The docs state what scripts may rely on.
- The docs state what scripts must not rely on.
- README and `docs/commands.md` link to the new output-format document.

## Phase 2: Move Progress To stderr

Change table-mode progress in `src/teamspeak_cli/main.cpp` so progress messages use `std::cerr` instead of `std::cout`.

Affected behavior:

- Final success output remains on stdout.
- Progress and diagnostics go to stderr.
- JSON and YAML output remain uncontaminated by progress.

Commands to cover:

- `connect`
- `disconnect`
- `update`
- `client start`
- `client stop`

Acceptance criteria:

- Tests prove progress does not appear on stdout.
- Tests prove final human output still appears on stdout.
- Existing JSON/YAML progress suppression still passes.

## Phase 3: Normalize Human Labels

Make detail labels readable in human output while keeping JSON keys unchanged.

Examples:

- `CurrentChannel` -> `Current channel`
- `SocketPath` -> `Socket path`
- `MediaTransport` -> `Media transport`
- `PIDFile` -> `PID file`
- `LogFile` -> `Log file`
- `QueuedPlaybackSamples` -> `Queued playback samples`
- `TransmitPathReady` -> `Transmit path ready`

Likely files:

- `src/teamspeak_cli/output/render.cpp`
- `src/teamspeak_cli/cli/command_router.cpp`
- `tests/output_render_test.cpp`
- `tests/cli_router_test.cpp`

Acceptance criteria:

- Human output uses readable labels consistently.
- JSON keys remain snake_case and unchanged.
- Renderer tests cover representative details blocks.

## Phase 4: Stabilize JSON As An API

Prefer documenting current JSON shapes first. Avoid adding a universal envelope until there is a concrete compatibility need.

Document:

- Top-level type for each command: object or array.
- Field names and types.
- Nullability.
- Whether fields are stable, best-effort, or debug-only.
- Error object shape.

Add snapshot-style or exact-structure tests for:

- `status`
- `plugin info`
- `channel list`
- `channel clients`
- `client list`
- `connect`
- `disconnect`
- `playback status`
- `events watch`
- structured errors

Acceptance criteria:

- Each documented JSON example is generated from or protected by tests.
- Debug-only fields stay hidden unless `--debug` is used.
- Error JSON remains structured and includes hints when available.

## Phase 5: Replace Or Restrict YAML

The current YAML renderer is custom and should either be replaced or clearly marked experimental.

Preferred path:

- Replace the custom YAML renderer with a real serializer.
- Add parser-backed tests that generated YAML can be read by a standard YAML parser.
- Fix sequence-of-map output so lists render idiomatically.

Short-term fallback:

- Mark YAML experimental in help and docs.
- Keep YAML out of the stable scripting contract until parser-backed tests exist.

Acceptance criteria:

- YAML generated by representative commands parses successfully.
- YAML string escaping handles quotes, backslashes, control characters, and multiline values safely.
- Docs state whether YAML is stable or experimental.

## Phase 6: Add Machine-Output Ergonomics

Add a way to select fields from JSON output.

Preferred first step:

- Add `--field <path>` for simple dot-path extraction.

Examples:

- `ts --json status --field phase`
- `ts --json plugin info --field media_diagnostics.transmit_path_ready`

Possible later step:

- Add `--jq <expr>` if a dependency strategy is acceptable.

Acceptance criteria:

- Field extraction works after command execution and does not complicate command handlers.
- Missing fields produce a usage or data error with a clear message.
- Extracted scalar output is easy to use in shell scripts.

## Phase 7: Improve Table Controls

Add table controls used by mature CLIs.

Initial options:

- `--no-headers`
- `--wide`

Candidate commands:

- `channel list`
- `client list`
- `channel clients`
- `events hook list`
- `message inbox`

Acceptance criteria:

- `--no-headers` removes only the header row.
- `--wide` adds useful extra columns without changing default table output.
- JSON output is unaffected by table options.

## Phase 8: Improve Help Text

Expand command metadata so generated help can include option descriptions, accepted values, defaults, and examples.

Likely changes:

- Extend `CommandDoc`.
- Replace plain option name lists with richer option metadata.
- Include output notes for scriptable commands.

Acceptance criteria:

- `ts <command> --help` explains each command-specific option.
- Defaults are visible where meaningful.
- Accepted values are visible for enum-like options.
- Existing contextual help behavior remains intact.

## Phase 9: Make Event Watching Stream-Friendly

Keep current batch output for compatibility, then add a streaming machine format.

Possible interface:

- `--output ndjson`
- or `ts events watch --stream --json`

Behavior:

- Emit one JSON object per event.
- Flush after every event.
- Keep table mode human-readable.

Acceptance criteria:

- NDJSON output can be consumed line by line.
- Batch JSON remains the default for `--json`.
- Tests cover at least two emitted events.

## Suggested Implementation Order

1. Document current output behavior.
2. Move progress to stderr.
3. Normalize human labels.
4. Stabilize and test JSON schemas.
5. Replace or restrict YAML.
6. Add field extraction.
7. Add table controls.
8. Improve help text.
9. Add streaming event output.

Phases 1 through 3 are low-risk polish and should happen first. Phases 4 and 5 define the machine-output contract. Phases 6 through 9 add power-user behavior after the foundations are stable.
